# 天织 ontop-aether 使用说明书

本文档说明 `ontop-aether` 的运行方式、核心工作流、页面操作、认证机制和部署形态。

## 1. 系统概述

`ontop-aether` 是一个基于 Ontop 的语义数据平台，把关系数据库映射成可查询、可发布、可供 AI 消费的虚拟知识图谱。

核心工作链路（四步）：

```
连接数据源 → 分析结构 → 生成与编辑映射 → 查询与问答
```

## 2. 双后端架构

前端 `server.ts` 根据路径自动路由到不同后端：

```
浏览器 → ontop-ui (Next.js)
           │
           ├── /api/v1/datasources (CRUD)      → ontop-engine (Java)
           ├── /api/v1/datasources/*/bootstrap  → ontop-backend (Python, 需 LLM)
           ├── /api/v1/endpoint-registry        → ontop-engine (Java)
           ├── /api/v1/mappings                 → ontop-engine (Java)
           ├── /api/v1/ontology                 → ontop-engine (Java)
           ├── /api/v1/sparql/*                 → ontop-engine (Java)
           ├── /api/v1/auth/*                   → ontop-backend (Python)
           ├── /api/v1/ai/*                     → ontop-backend (Python)
           ├── /api/v1/annotations/*            → ontop-backend (Python)
           ├── /api/v1/glossary/*               → ontop-backend (Python)
           ├── /api/v1/suggestions/*            → ontop-backend (Python)
           ├── /api/v1/governance/*             → ontop-backend (Python)
           └── /api/v1/publishing/*             → ontop-backend (Python)
```

| 服务 | 职责 | 技术栈 |
|------|------|--------|
| ontop-engine | 数据源 CRUD、端点切换、映射读写、TTL 解析、SPARQL 代理 | Spring Boot + JdbcTemplate + OWLAPI |
| ontop-backend | Bootstrap 编排、AI/LLM、标注、词汇表、精化、治理、认证 | FastAPI + OpenAI SDK |
| ontop-endpoint | SPARQL 查询执行、SQL reformulate | Ontop 5.5.0 |

SQLite 共享（WAL 模式）：
- Java 管理：datasources, endpoint_registry, query_history
- Python 管理：ai_config, annotations, glossary, users, sessions, suggestions, publishing, 治理表
- 密码 Fernet 加密（Java/Python 零迁移兼容）

## 3. 部署与启动

### 3.1 Docker Compose

在 `ontop-aether/` 目录下执行。

**LVFA / Mondial 演示环境**（推荐）

```bash
docker compose -f docker-compose.lvfa.yml up -d --build
```

| 服务 | 地址 |
|------|------|
| 前端 | http://localhost:3001 |
| 后端 API 文档 | http://localhost:8001/docs |
| Java 引擎 | http://localhost:8083 |
| SPARQL Endpoint | http://localhost:18081 |
| PostgreSQL | localhost:5436 (admin/test123) |

**默认 retail 环境**

```bash
docker compose up -d --build
```

前端 http://localhost:3000，后端 :8000，引擎 :8081，端点 :18080。

**MySQL 演示环境**

```bash
docker compose -f docker-compose.mysql.yml up -d --build
```

前端 :3002，后端 :8002，引擎 :8084，端点 :18082。

### 3.2 本地开发

Java 引擎：
```bash
cd ontop-engine && mvn spring-boot:run
# 环境变量: SQLITE_DB_PATH, ENCRYPTION_KEY_PATH, ONTOP_ENDPOINT_URL
```

Python 后端：
```bash
cd ontop-backend
pip install -r requirements.txt
python3 -m uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

前端：
```bash
cd ontop-ui && pnpm install && pnpm dev
# 环境变量: BACKEND_URL=http://localhost:8000, ENGINE_URL=http://localhost:8081
```

## 4. 认证

### 4.1 登录

- 地址：`/login`
- 默认账号：`admin` / `admin123`
- 登录后 token 存 localStorage/sessionStorage
- 未登录访问业务页自动跳转 `/login`

### 4.2 双轨认证

- 密码登录：Web 界面使用
- API Key（`oak_` 格式）：外部 API 调用使用，在「平台治理 → 访问控制」管理

### 4.3 认证接口

| 接口 | 说明 |
|------|------|
| `POST /api/v1/auth/login` | 登录，返回 Bearer token |
| `GET /api/v1/auth/me` | 校验当前会话 |
| `POST /api/v1/auth/logout` | 退出 |
| `POST /api/v1/auth/change-password` | 修改密码 |

## 5. 推荐工作流

### 步骤 1：连接数据源 `/datasource`

1. 点击「添加数据源」
2. 填写 JDBC 信息：
   - 名称（如 `Mondial PostgreSQL`）
   - JDBC URL（如 `jdbc:postgresql://postgres:5432/mondial_db`）
   - 用户名 / 密码
   - 驱动（默认 `org.postgresql.Driver`）
3. 点击「测试连接」确认连通
4. 保存

### 步骤 2：分析结构 `/db-schema`

1. 选择数据源
2. 浏览表结构：表名、列名、类型、主键、外键
3. 了解表间关系，为 Bootstrap 做准备

### 步骤 3：Bootstrap 与映射

1. 回到 `/datasource`，对目标数据源点击「Bootstrap」
2. LLM 分析数据库结构，自动生成本体（.ttl）+ 映射规则（.obda）
3. 进入 `/mapping` 浏览、编辑映射规则
4. 点击「验证」检查映射正确性
5. 点击「重启端点」应用更改

### 步骤 4：查询与问答

- **SPARQL 查询** `/sparql`：直接编写 SPARQL，查看结果和 SQL reformulate
- **AI 助手** `/ai-assistant`：用自然语言提问，LLM 自动生成 SPARQL 并执行

## 6. 页面详细说明

### 6.1 首页 `/`

平台入口页面，展示：
- 端点运行状态（绿色/红色标识）
- 统计概览：数据源数量、映射文件数、本体类数、查询历史数
- 核心工作区卡片（6 个快速入口）
- 推荐工作流（四步指引）
- 当前 AI 模型与 Provider 配置

### 6.2 数据源管理 `/datasource`

**操作**：
- 列表查看所有数据源
- 添加/编辑/删除数据源
- 测试连接（Java 进程内调用 Ontop，无 HTTP 开销）
- 查看 Schema 列表和详细结构
- 发起 Bootstrap（路由到 Python 后端调用 LLM）

**注意**：CRUD 操作由 Java 引擎处理，Bootstrap 由 Python 后端处理。

### 6.3 数据库概览 `/db-schema`

**操作**：
- 下拉选择数据源
- 浏览所有表、列、主键、外键
- 搜索表名
- 查看列详情（类型、是否可空、默认值）

### 6.4 SPARQL 查询 `/sparql`

**操作**：
- 在编辑器中编写 SPARQL 查询
- 点击执行，结果以表格展示
- 支持 JSON / XML / CSV / Turtle 格式
- 查看 SQL reformulate（Ontop 重写的原生 SQL）
- 查看查询历史，一键重跑或删除

**后端链路**：前端 → Java 引擎 → Ontop Endpoint → PostgreSQL

### 6.5 映射编辑 `/mapping`

**操作**：
- 左侧列出所有 .obda 文件
- 选中文件查看映射规则列表（Mapping ID / Target / Source）
- 编辑映射内容并保存（Java 端 OBDA 序列化）
- 验证映射正确性
- 重启端点应用更改

### 6.6 AI 助手 `/ai-assistant`

**操作**：
- 在聊天框输入自然语言问题（如"有哪些湖泊面积超过 1000 平方公里？"）
- LLM 自动生成 SPARQL → 执行 → 流式返回结果
- 可查看生成的 SPARQL 和重写后的 SQL
- 使用快捷问题快速提问

**前提**：需在 `/settings` 配置 LLM Provider。

### 6.7 本体可视化 `/ontology`

**操作**：
- 选择 .ttl 文件
- 查看解析后的 OWL 类、对象属性、数据属性
- Vis.js 关系图谱（可缩放、拖拽）
- Tab 切换：图谱视图 / 定义列表

### 6.8 语义标注 `/annotations`

**操作**：
- 页面默认加载当前激活数据源的标注
- 下拉可切换查看其他数据源
- 三 Tab 管理：待审核 / 已接受 / 已拒绝
- 逐条操作：接受 / 拒绝 / 编辑（人工覆盖 LLM 标注）
- 批量操作：全选接受/拒绝
- 「合并到本体」：将 accepted 标注写入 active_ontology.ttl

**数据流**：Bootstrap 后 LLM 自动生成 → 人工审核 → 合并到本体

### 6.9 业务词汇表 `/glossary`

**操作**：
- 维护业务口语词 → 本体属性/类 URI 映射
- LLM 自动从已审核标注推导词汇
- 人工添加/编辑词汇条目（永不被 LLM 覆盖）
- 全局词汇（ds_id 为空）跨数据源共享
- 导出/导入 JSON
- 模糊搜索，按类型过滤

**作用**：AI 查询时按关键词匹配 Top-12 词汇注入 SPARQL Prompt，提升查询准确率。

### 6.10 本体精化 `/refinement`

**操作**：
- 点击「AI 分析」对当前本体生成改进建议
- 6 种建议类型：
  - `RENAME_CLASS` / `RENAME_PROPERTY` — 重命名
  - `REFINE_TYPE` — 类型精化
  - `ADD_LABEL` — 添加标签
  - `ADD_SUBCLASS` — 添加子类（高风险，给操作指引）
  - `MERGE_CLASS` — 合并类（高风险，给操作指引）
- 低风险类型支持一键「应用」自动修改 TTL
- 批量应用所有已接受的可自动建议

### 6.11 AI 设置 `/settings`

**操作**：
- 选择 LLM Provider（OpenAI / LM Studio / Ollama / DeepSeek / 智谱 / Azure / Anthropic / 自定义）
- 输入 API 地址和 Key
- 自动拉取可用模型列表
- 编辑系统提示词（支持模板变量）
- 管理快捷问题

### 6.12 数据发布 `/publishing`

**操作**：
- **API 接入**：SPARQL 端点健康检查、API Key 生成/管理、CORS 配置
- **MCP 服务**：一键启停 MCP Server（Streamable HTTP）、工具列表自动推导、目标平台配置片段生成（Claude Desktop / Cursor / Windsurf）
- **工具定义**：四种格式一键生成与预览（OpenAI Function Calling / Anthropic Tool Use / OpenAPI 3.0 / Generic JSON Schema）

### 6.13 平台治理 `/governance`

三个子页面：

**项目管理** `/governance/projects`
- 创建/管理项目
- 多租户隔离
- 固定环境：dev / test / prod

**访问控制** `/governance/access`
- RBAC：8 个内置角色 + 16 个权限
- 角色绑定（用户 → 角色 → 项目/环境）
- API Key 管理（`oak_` 格式，支持权限范围限定）

**审计日志** `/governance/audit`
- 自动记录所有 HTTP 请求
- 按时间、操作者、资源类型、状态筛选
- 查看请求详情

### 6.14 系统设置 `/system`

- 当前用户信息
- 后端服务健康检查
- Ontop 端点运行状态
- 系统配置只读展示

## 7. 端点注册表与多 Repository 管理

端点注册表管理 Ontop SPARQL 端点中的多个 Repository 实例。

**多 Repository 架构**：

- 每个完成 Bootstrap 的数据源会注册为独立的 Repository
- 所有 Repository 同时可查询，通过 `/{dsId}/sparql` 路由
- `/sparql` 无参路由使用当前激活数据源（向后兼容）

**激活数据源**：

1. 进入 `/datasource`，对目标数据源点击「激活」
2. 系统通过 Repository 管理 API 确保数据源已注册
3. 调用 `PUT /api/v1/repositories/{dsId}/activate` 设为默认
4. 无需重启端点，切换时间接近零

**影响范围**：
- `/{dsId}/sparql` 路由始终可用，不受激活状态影响
- `/sparql` 无参路由指向当前激活数据源
- 标注/词汇/精化页面默认跟随激活数据源

## 8. 数据存储

### 8.1 SQLite 元数据库

位置：`ontop-backend/data-*/ontop_ui.db`（Docker 中 `/app/data/ontop_ui.db`）

主要表：

| 表 | 管理方 | 说明 |
|----|--------|------|
| datasources | Java | 数据源连接信息（密码 Fernet 加密） |
| endpoint_registry | Java | 端点注册与激活状态 |
| query_history | Java | SPARQL 查询历史 |
| users | Python | 用户账号 |
| sessions | Python | 登录会话 token |
| ai_config | Python | LLM 配置 |
| semantic_annotations | Python | 语义标注 |
| business_glossary | Python | 业务词汇表 |
| ontology_suggestions | Python | 本体精化建议 |
| publishing_config | Python | 发布配置 |
| 治理相关表 | Python | 租户/项目/角色/审计等 |

### 8.2 文件系统产物

位置：`ontop-backend/data-*/{ds_id}/`

| 文件 | 说明 |
|------|------|
| `*_ontology.ttl` | OWL 本体文件 |
| `*_mapping.obda` | OBDA 映射文件 |
| `active/merged_ontology.ttl` | 合并标注后的激活本体 |
| `manifest.json` | Bootstrap 元数据 |
| `selected_tables.json` | 选表记录 |

共享激活目录：`ontop-endpoint/{env}-active/`

| 文件 | 说明 |
|------|------|
| `active_ontology.ttl` | 当前端点使用的本体 |
| `active_mapping.obda` | 当前端点使用的映射 |
| `active.properties` | 当前端点 JDBC 配置 |

## 10. 当前限制

- 认证为基础账号体系，非企业级 IAM / SSO
- 部分逻辑仍依赖 active endpoint（如标注合并）
- 平台治理处于基础阶段，尚未与外部 IAM 集成
- 文档截图可能滞后于 UI，以代码和运行页面为准
