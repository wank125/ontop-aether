# Ontop 技术路线分析与测试总结

> 基于零售数据实测，对比分析数据库→AI 的三条技术路线
> 测试时间：2026年3月

---

## 一、Ontop 的核心作用

**给关系型数据库加一层"语义映射"，让用户用 SPARQL 查询数据库，而无需写 SQL。**

1. **自动生成语义层**：一条 `bootstrap` 命令扫描数据库 schema，自动生成 OWL 本体 + 映射规则
2. **SPARQL → SQL 实时翻译**：用户写 SPARQL，Ontop 实时翻译成 SQL 发给数据库执行
3. **虚拟知识图谱（VKG）**：用户看到统一的知识图谱视图，数据始终在数据库里

**不做什么**：不存储数据、不做图遍历、不做自然语言交互、不做推理（仅 OWL 2 QL）

---

## 二、数据库 → AI 的三条技术路线

### 路线对比

```
路线 1（最简单）: 数据库 ──────────────→ LLM（Text-to-SQL）
路线 2（语义层）: 数据库 → Ontop ────→ LLM（Text-to-SPARQL）
路线 3（图遍历）: 数据库 → Neo4j ────→ LLM（Text-to-Cypher）
```

### 选择依据

| 场景 | 推荐路线 | 理由 |
|------|---------|------|
| 单个数据库，LLM 查询 | 路线 1 Text-to-SQL | 最简单，链路最短 |
| 需要图遍历、关系探索 | 路线 3 Text-to-Cypher | Neo4j 擅长图操作 |
| 多数据源统一语义 | 路线 2 Ontop | 语义映射层的核心价值 |
| 需要 SPARQL 标准接口 | 路线 2 Ontop | W3C 标准化查询 |
| 需要本体推理 | 路线 2 Ontop | OWL 2 QL 内置 |

## 三、Ontop 映射逻辑

### 映射的本质

**把 SQL 查询结果 → RDF 三元组的模板：**

```
映射规则 = SQL 查询（source） + RDF 模板（target）
```

### 两类映射规则

**第一类：表 → 类 + 属性**

```
source: SELECT * FROM "dim_store"
target: <.../dim_store/store_id={store_id}> a <.../dim_store> ;
        <...#store_id> {store_id} ;
        <...#name> {name} ;
        <...#region> {region} .
```

**第二类：外键 → 关系**

```
source: SELECT e.emp_id, s.store_id
        FROM dim_employee e, dim_store s
        WHERE e.store_id = s.store_id        ← JOIN
target: <.../emp_id={emp_id}> <...#ref-store_id> <.../store_id={store_id}> .
```

### Bootstrap 自动推导原理

```
数据库元数据                    映射规则
──────────                    ──────────
表 (3张)          →  3条类映射
主键 (3个)        →  URI 模板构造
列 (12个)         →  属性映射（嵌入类映射）
外键 (3个)        →  3条关系映射（JOIN → 对象属性）
```

推导规则：
- 表 → OWL Class
- 主键 → URI 模板
- 列 → DatatypeProperty（类型自动映射：integer→xsd:integer, varchar→xsd:string）
- 外键 → ObjectProperty（自动生成 JOIN 查询）

## 四、Text-to-SQL 实测结果

### 测试配置

- 模型：LM Studio 本地 `glm-4.7-flash`
- 数据库：PostgreSQL (retail_db)
- 方式：自然语言 → SQL → 查询 → 自然语言回答

### 测试结果（7题，6/7 正确）

| 问题 | 生成的 SQL | 回答 | 正确？ |
|------|-----------|------|--------|
| 有哪些门店？ | `SELECT name FROM dim_store` | 华东旗舰店、南京中心店、广州天河店、深圳南山店 | ✅ |
| 华东旗舰店有多少员工？ | `SELECT COUNT(*) FROM dim_store s JOIN dim_employee e ON ... WHERE s.name='华东旗舰店'` | 2名 | ✅ |
| 哪个门店销售额最高？ | `SELECT name, SUM(amount) ... GROUP BY name ORDER BY total DESC LIMIT 1` | 华东旗舰店 ¥4700 | ✅ |
| 李四经手的销售总额？ | `SELECT SUM(amount) FROM fact_sales JOIN dim_employee ON ... WHERE name='李四'` | ¥4700 | ✅ |
| 华南地区的门店有哪些员工？ | `SELECT ... JOIN ... WHERE region='华南'` | 赵六（店长）、钱七（销售员） | ✅ |
| 3月每天销售趋势？ | SQL 正确但加了多余 WHERE 条件 | 空 | ⚠️ |
| 哪些员工没有销售记录？ | `LEFT JOIN ... WHERE sale_id IS NULL` | 张三 | ✅ |

## 五、Text-to-Cypher 实测结果

### 测试配置

- 模型：LM Studio 本地 `glm-4.7-flash`
- 图库：Neo4j 5
- 管线：PostgreSQL → Ontop materialize → RDF → Python → Neo4j

### 测试结果（5题，5/5 正确）

| 问题 | 生成的 Cypher | 回答 | 正确？ |
|------|-------------|------|--------|
| 有哪些门店？ | `MATCH (s:dim_store) RETURN s.name` | 4家门店 | ✅ |
| 华东旗舰店有多少员工？ | `MATCH (e)-[:REF_STORE_ID]->(s{name:'华东旗舰店'}) RETURN count(e)` | 2名 | ✅ |
| 哪个门店销售额最高？ | `MATCH (f)-[:REF_STORE_ID]->(s) RETURN s.name, sum(f.amount) ORDER BY ... DESC LIMIT 1` | 华东旗舰店 ¥4700 | ✅ |
| 李四经手的销售总额？ | `MATCH (e{name:'李四'})<-[:REF_EMP_ID]-(f) RETURN sum(f.amount)` | ¥4700 | ✅ |
| 华南地区门店员工？ | `MATCH (e)-[:REF_STORE_ID]->(s{region:'华南'}) RETURN e.name` | 赵六、钱七 | ✅ |

### Prompt 优化经验

1. **关系方向必须明确**：`(dim_employee)-[:REF_STORE_ID]->(dim_store)`，不能用自然语言描述
2. **提供数据示例**：包含实际数据值（如 region='华南' 不是 '华南地区'）
3. **提供 Cypher 示例**：few-shot 比纯 schema 描述更有效
4. **推理模型不适用**：qwen3.5-27b、deepseek-r1 等推理模型把内容放在 `reasoning_content`，`content` 为空

## 六、数据库直接到 Neo4j（跳过 Ontop）

### 三种方式

| 方式 | 工具 | 特点 |
|------|------|------|
| APOC JDBC | `apoc.load.jdbc` | 在 Cypher 里直接查数据库，支持增量同步 |
| Neo4j ETL Tool | 图形化工具 | 自动识别表→节点、外键→关系 |
| Python 直连 | psycopg2 + neo4j | 最灵活，完全可控 |

### APOC JDBC 示例

```cypher
// 直接从 PostgreSQL 读数据创建节点
CALL apoc.load.jdbc('jdbc:postgresql://localhost:5433/retail_db?user=admin&password=test123',
                    'SELECT * FROM dim_store')
YIELD row
CREATE (s:dim_store {store_id: row.store_id, name: row.name, region: row.region})
```

**结论：如果目标只是把数据库数据导入 Neo4j，完全不需要 Ontop。**

## 七、多数据源统一语义层（Ontop 的核心价值场景）

### 问题：同一个概念，分散在不同系统

```
ERP (MySQL)              CRM (PostgreSQL)          数仓 (SQL Server)
t_customer               clients                   dim_user
 cust_id (INT)           client_uuid (UUID)        user_id (INT)
 cust_name               full_name                 display_name
 city                    region_code               province
 phone                   contact_email             mobile
```

同一个"客户"，三个系统三种叫法、三种结构。

### Ontop 的解决方案

**定义一次本体，每个数据源写一条映射：**

```obda
# ERP → Customer
source:  SELECT cust_id, cust_name, city FROM t_customer
target:  <.../customer/{cust_id}> a :Customer ; :name {cust_name} ; :region {city} .

# CRM → Customer（同一个类）
source:  SELECT client_uuid, full_name, region_code FROM clients
target:  <.../customer/{client_uuid}> a :Customer ; :name {full_name} ; :region {region_code} .

# 数仓 → Customer（同一个类）
source:  SELECT user_id, display_name, province FROM dim_user
target:  <.../customer/{user_id}> a :Customer ; :name {display_name} ; :region {province} .
```

用户只需一条 SPARQL，Ontop 自动翻译成多条 SQL 去各系统查询：

```sparql
SELECT (COUNT(?c) AS ?total) WHERE {
  ?c a :Customer ; :region "华东"
}
```

### 价值总结

| 维度 | 没有统一语义层 | 有 Ontop |
|------|--------------|---------|
| 用户学习成本 | N 套 SQL、N 套表名 | 1 套 SPARQL、1 套本体 |
| 加新数据源 | 重写所有查询 | 只加一条映射规则 |
| 字段改名 | 改所有下游 SQL | 只改映射规则 |
| 语义一致性 | cust_name vs full_name 谁对？ | 统一叫 :name |

---

## 八、最终结论

1. **单数据源 + LLM 查询**：Text-to-SQL 最简单，不需要 Ontop 也不需要 Neo4j
2. **需要图遍历**：直连 Neo4j（APOC/Python），不需要 Ontop
3. **多数据源统一语义**：这是 Ontop 的真正价值场景
4. **本地模型可用**：glm-4.7-flash 在 Text-to-SQL 和 Text-to-Cypher 上都能正确生成查询
5. **Ontop 的定位**：不是数据搬运工具，而是跨数据源的"翻译器 + 统一词典"
